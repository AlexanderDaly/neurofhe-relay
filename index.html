Bio-Digital Privacy

NeuroFHE Relay

A privacy layer for event-driven AI: neuromorphic systems make data sparse, FHE keeps the sensitive computation private.

01 / Problem

Edge AI is entering places where raw data should not travel.

Wearables, event cameras, robotics, smart buildings, and industrial monitoring all create sensitive continuous signals. The bio-digital boundary should keep raw signals local while allowing useful event intelligence to cross under explicit privacy controls.

02 / Insight

Do not make FHE digest the whole world.

Neuromorphic systems turn signals into sparse spikes and events. That representation is a better starting point for encrypted inference than dense frames or large tensors.

NeuromorphicSparse, event-driven, low activity.
FHEPrivate compute over encrypted data.
BridgeEncode smaller workloads before encryption.
AvoidClaiming current neuromorphic chips directly accelerate full FHE.
03 / Architecture

A hybrid pipeline with clear boundaries.

The edge sees raw data. The compute layer sees ciphertext. The keyholder sees the final result.

1. Event source
Sensor, SNN, or synthetic event stream.
then
2. Spike encoder
Binary spikes, counts, low-bit windows.
then
3. FHE kernel
Encrypted accumulation, threshold, score.
then
4. Client decrypts
Only the final result leaves ciphertext.
04 / Octra Fit

Octra is the encrypted-compute lane, not the whole story.

Octra positions itself as an FHE blockchain network with isolated execution environments and HFHE. NeuroFHE Relay should first test and bound compact encrypted event inference locally, then map the right pieces into Octra programs or proof/state flows.

  • Use mature HE libraries for first benchmarks.
  • Use Octra for a feasibility adapter after operation shape is known.
  • Keep token or network dependence out of the first demo.
05 / First Demo

A tiny encrypted event benchmark now runs locally.

The current research-alpha scaffold classifies a synthetic 8 by 8 event window through a fixed linear model and toy additive HE path, with plaintext and encrypted scores matching. It is deliberately small, research-alpha, and honest about what the compute side can still see.

  • Run: `npm run demo`, `npm run benchmark`, and `npm test`.
  • Scores: normal 9, anomaly 51, classification anomaly.
  • Linear contract: scores = W x + bias, matrix shape 2 by 64.
  • Sparse path: 36 scalar multiplies/adds versus 128 for dense encrypted tensor scoring.
  • Benchmark emits privacy boundary, crypto inventory, ciphertext bytes, and dense baseline comparison.
  • Caveat: public active positions may reveal sparsity and timing metadata.

07 / 90-Day Plan

Make one evidence package strong enough to decide the project.

  • Month 1: plaintext event encoder and tiny SNN baseline.
  • Month 2: encrypted inference kernel, benchmark table, and crypto inventory.
  • Month 3: post-quantum envelope, visual demo, Octra feasibility note, and review package.

Decision rule: if sparse event representation does not materially reduce encrypted operation cost, pivot before overbuilding.
